# GRPO 学习笔记：从直觉到训练流程

[TOC]

GRPO，全称 Group Relative Policy Optimization，可以理解为：

> 对同一个 prompt 生成多个回答，然后用这一组回答内部的相对好坏来更新模型。

它的核心思想是：

```text
同一道题生成 G 个答案
→ 每个答案打 reward
→ 看每个答案比同组平均水平好还是差
→ 提高好答案的概率
→ 降低差答案的概率
→ 用 old policy 和 ref model 控制训练别跑偏
```

GRPO 常用于 LLM 的推理强化训练，比如数学、代码、逻辑题等 RLVR 场景。

---

## 1. GRPO 解决什么问题？

传统 PPO 训练 LLM 时，通常需要一个 value model / critic model 来估计 baseline。

大致是：

```text
reward - value = advantage
```

但 value model 很占显存，也增加训练复杂度。

GRPO 的做法是：

```text
不训练 critic / value model
而是让同一个 prompt 生成多个回答
用这一组回答的平均 reward 当 baseline
```

所以 GRPO 的重点是：

```text
用 group 内部的相对比较，替代 critic model。
```

---

## 2. GRPO 需要哪些角色？

GRPO 里容易混淆的模型主要有三个：

```text
policy model
old policy
ref model
```

它们的作用不同。

---

### 2.1 Policy model：正在训练的模型

Policy model 是当前正在被优化的 LLM。

记作：

```text
πθ
```

它负责生成答案，也会被梯度更新。

例如：

```text
Qwen2.5-7B-Instruct
DeepSeekMath-Instruct
Llama-Instruct
```

训练完成后，我们真正得到的就是更新后的 policy model。

---

### 2.2 Old policy：生成当前 batch 数据时的 policy 快照

Old policy 不是另一个独立模型。

它就是：

```text
生成这批训练样本时的 policy 快照
```

比如当前 policy 是 M5，它生成了一批答案。

开始更新后，policy 可能变成 M6。

这时候：

```text
old policy = M5
current policy = M6
```

old policy 的作用是提供一个短期参照：

```text
当前 policy 相比生成这些答案时的 policy，到底变化了多少？
```

因此它会用于计算：

```text
π_new(output | prompt) / π_old(output | prompt)
```

这个比例叫 policy ratio。

---

### 2.3 Ref model：初始模型 / 长期锚点

Ref model 通常是 RL 训练开始前的 SFT 模型。

它一般固定不动。

例如：

```text
M0 = SFT 后的模型
```

训练过程中：

```text
ref model = M0，一直不变
policy model = M1, M2, M3, ... 不断变化
old policy = 每一轮生成数据时的 policy 快照
```

ref model 的作用不是生成训练数据，而是作为长期锚点，防止 policy 为了追 reward 偏离原始模型太远。

---

## 3. 三者的关系

训练刚开始时，三者可能是一样的：

```text
policy model = M0
old policy   = M0
ref model    = M0
```

训练一段时间后：

```text
ref model    = M0
old policy   = M5
policy model = M6
```

也就是说：

```text
ref model 固定不动
old policy 跟着训练过程不断变化
policy model 是当前正在更新的模型
```

一句话总结：

```text
policy model：正在训练的人
old policy：刚刚生成这批答案时的人
ref model：最初那个不能偏离太远的基准人
```

---

## 4. 为什么不用 ref model 直接生成回答？

一个常见疑问是：

> 为什么不直接用 ref model 根据 prompt 生成答案，然后 reward 打分，再训练 policy？

这样当然可以做，但它不叫标准 GRPO，更像：

```text
rejection sampling
best-of-N distillation
filtered SFT
RFT
offline RL
```

因为 ref model 是固定的旧模型。

如果一直用 ref model 生成数据，那么 policy 学到的是：

```text
旧模型生成的一堆回答里，哪些相对更好
```

但 GRPO 想做的是：

```text
当前 policy 自己生成一组回答
然后看当前 policy 的哪些行为更好
再更新当前 policy
```

强化学习的关键是：

```text
谁要被训练，就让谁产生行为；
然后根据这些行为的结果来更新它。
```

如果 policy 已经从 M0 训练到了 M5，还一直让 M0 生成数据，那么这些数据已经不能很好反映 M5 当前的能力、错误和行为习惯。

---

## 5. 为什么需要 old policy？

另一个常见疑问是：

> 既然用 policy 生成答案，为什么还要 old policy？

答案是：

```text
old policy 就是生成答案时的 policy。
```

它存在的原因不是为了替代 policy 生成，而是为了在更新时提供参照。

假设 M5 生成了一个回答：

```text
prompt -> output
```

生成时这个 output 的概率是：

```text
π_M5(output | prompt)
```

随后我们更新模型，policy 变成 M6。

现在这个 output 在 M6 下的概率变成：

```text
π_M6(output | prompt)
```

训练时需要知道：

```text
M6 相比 M5，对这个 output 的概率提高或降低了多少？
```

所以要计算：

```text
ratio = π_M6(output | prompt) / π_M5(output | prompt)
```

这里的 M5 就是 old policy。

old policy 的作用是：

```text
限制 policy 每次更新不要太猛。
```

---

## 6. Old policy 和 ref model 都是锚，但锚定对象不同

可以这样记：

```text
old policy：短期锚
ref model：长期锚
```

old policy 控制的是：

```text
这一步更新不要迈太大
```

ref model 控制的是：

```text
整体不要离初始模型太远
```

具体区别：

```text
old policy 用于 policy ratio / clipping
ref model 用于 KL penalty
```

也就是：

```text
old policy 负责局部稳定
ref model 负责全局稳定
```

---

## 7. GRPO 的基本训练流程

一轮 GRPO 大致如下：

```text
1. 从训练集中采样一批 prompts
2. 当前 policy 的快照生成每个 prompt 的 G 个回答
3. reward function / verifier 给每个回答打分
4. 同一个 prompt 下，对 G 个 reward 做组内归一化
5. 得到每个回答的 advantage
6. 用 advantage 更新 policy
7. 用 old policy ratio / clipping 限制更新幅度
8. 用 ref model 的 KL penalty 约束整体漂移
9. 进入下一轮
```

更直观地写：

```text
prompt
  ↓
policy / old policy 生成 G 个回答
  ↓
reward function 打分
  ↓
组内比较，算 advantage
  ↓
更新 policy
  ↓
old policy 控制一步别迈太大
  ↓
ref model 控制整体别跑太远
```

---

## 8. 多组数据是怎么生成的？

GRPO 的原始数据通常只需要 prompt，以及 reward 需要的标准答案或验证信息。

例如数学题数据：

```json
{
  "prompt": "小明有 3 个苹果，又买了 2 个，一共有几个？",
  "answer": "5"
}
```

训练时，模型会对同一个 prompt 采样多个 completion。

例如设置：

```text
num_generations = 4
```

那么同一个问题会生成 4 个回答：

```text
回答 1：3 + 2 = 5，所以答案是 5。
回答 2：小明一共有 6 个苹果。
回答 3：一共有 5 个。
回答 4：答案是 4。
```

这些回答不是人工提前写好的，而是 policy model 在线生成的。

因此所谓 group 是：

```text
同一个 prompt 下，由当前 policy 采样出来的一组回答。
```

---

## 9. Reward 怎么算？

Reward 可以来自规则函数、verifier，也可以来自 reward model。

在数学任务中，常见 reward 是：

```text
最终答案是否正确
格式是否符合要求
推理过程是否包含指定结构
长度是否合理
```

例如：

```text
回答 1：答案 5，reward = 1
回答 2：答案 6，reward = 0
回答 3：答案 5，reward = 1
回答 4：答案 4，reward = 0
```

那么这个 group 的 reward 是：

```text
[1, 0, 1, 0]
```

---

## 10. Advantage 是什么？

Advantage 可以理解成：

```text
这个回答比同组平均水平好多少？
```

GRPO 的 advantage 来自 group 内部比较。

假设同一个 prompt 生成 4 个回答，reward 是：

```text
[1, 1, 0, 0]
```

组内平均 reward 是：

```text
mean = 0.5
```

如果标准差约为 0.5，那么每个回答的 advantage 大致是：

```text
[+1, +1, -1, -1]
```

意思是：

```text
reward 高于组均值 → positive advantage → 提高概率
reward 低于组均值 → negative advantage → 降低概率
```

所以 GRPO 不是简单看绝对 reward，而是看：

```text
同一个 prompt 下，哪些回答相对更好？
```

这就是 Group Relative 的含义。

---

## 11. 为什么需要 group？

如果一个 prompt 只生成一个回答，reward 是 1 或 0，那么很难知道它相对当前模型水平是好还是坏。

GRPO 需要多个回答，是为了做组内比较。

例如：

```text
同一道题生成 8 个答案
其中 3 个正确，5 个错误
```

这时训练信号很清楚：

```text
提高正确答案的概率
降低错误答案的概率
```

如果所有回答都对：

```text
[1, 1, 1, 1]
```

那组内没有区分度。

如果所有回答都错：

```text
[0, 0, 0, 0]
```

也没有区分度。

最有学习价值的是：

```text
同组里有好有坏
```

---

## 12. GRPO 的 loss 直觉

不用先背公式，只要记住直觉：

```text
如果某个回答的 reward 高于同组平均：
    提高这个回答中 token 的概率

如果某个回答的 reward 低于同组平均：
    降低这个回答中 token 的概率

但更新不能太猛：
    用 old policy ratio / clipping 控制

整体不能跑太偏：
    用 ref model 的 KL 控制
```

GRPO 大概在做：

```text
maximize group-relative reward
while keeping policy update stable
and keeping policy close enough to ref model
```

---

## 13. 什么是 KL？

KL，全称 Kullback-Leibler divergence。

简单理解：

```text
KL = 衡量两个概率分布有多不一样
```

在 LLM 里，就是衡量：

```text
当前 policy 对下一个 token 的概率分布
和
ref model 对下一个 token 的概率分布
差了多少
```

如果两个模型对 token 的概率判断完全一样：

```text
KL = 0
```

如果差异越大：

```text
KL 越大
```

---

## 14. 用例子理解 KL

假设同一个 prompt 下，下一个 token 有三个可能：

```text
ref model 认为：
"A" 概率 40%
"B" 概率 30%
"C" 概率 30%

policy model 认为：
"A" 概率 90%
"B" 概率 5%
"C" 概率 5%
```

这说明 policy 已经和 ref 差得比较远。

KL 就是用来量化这种差异。

直觉上：

```text
policy 和 ref 越像，KL 越小
policy 和 ref 越不像，KL 越大
```

---

## 15. 为什么需要 KL？

因为 reward 往往是不完整的。

例如数学 RL 里，reward 可能只看最终答案是否正确：

```text
答案正确 → reward = 1
答案错误 → reward = 0
```

但这个 reward 不一定关心：

```text
语言是否自然
推理是否清晰
是否遵循指令
是否保持通用对话能力
是否出现奇怪格式
是否 reward hacking
```

如果没有 KL，模型可能为了拿高 reward 学出奇怪策略。

例如：

```text
只输出裸答案
过度模板化
输出格式越来越怪
语言质量下降
泛化能力下降
为了骗 verifier 学会投机
```

KL 的作用是：

```text
你可以为了更高 reward 改进，
但不要离原来的 SFT/ref 模型太远。
```

---

## 16. KL 在训练目标里的位置

GRPO / PPO 类方法常见目标可以直觉写成：

```text
maximize reward - beta * KL(policy || ref)
```

意思是：

```text
想让 reward 更高
但如果 policy 离 ref 太远，就扣分
```

其中 beta 是 KL 惩罚强度。

```text
beta 大：
    模型更保守，更像 ref，训练更稳，但提升可能慢

beta 小：
    模型更自由，更容易追 reward，但也更容易跑偏
```

所以 KL 本质上是一种正则化。

---

## 17. 一句话复述 GRPO

GRPO 就是：

> 让当前模型对同一个问题生成多个答案，用 reward 判断这些答案在组内谁更好，然后提高好答案概率、降低差答案概率；同时用 old policy 控制每次更新别太猛，用 ref model 的 KL 控制整体别偏离原始模型太远。
