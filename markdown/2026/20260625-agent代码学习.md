## 2. OpenAI vs Anthropic 消息格式对比

### 2.1 核心差异

| 维度 | OpenAI | Anthropic |
|------|--------|-----------|
| **工具调用字段** | `tool_calls` 是 message 的顶层字段 | tool_use block 在 content 列表里 |
| **工具结果** | `role: "tool"` + `tool_call_id` | `role: "user"` + `type: "tool_result"` block |
| **参数格式** | JSON 字符串 `arguments: '{"city":"上海"}'` | 直接 dict `input: {"city": "上海"}` |
| **System 消息** | 在 messages 数组里 `role: "system"` | 独立参数，不在 messages 里 |
| **多模态** | `content` 可以是字符串或数组 | `content` 始终是数组 |
| **Finish Reason** | `finish_reason: "tool_calls"` | `stop_reason: "tool_use"` |

### 2.2 示例对比

```python
# ===== Anthropic 格式 =====
# Assistant 消息：文本 + 工具都在 content 里
{
    "role": "assistant",
    "content": [
        {"type": "text",      "text": "我来查一下"},
        {"type": "tool_use",  "name": "get_weather", "input": {"city": "上海"}, "id": "call_1"},
        {"type": "tool_use",  "name": "get_weather", "input": {"city": "北京"}, "id": "call_2"},
    ]
}

# Tool 结果：user 消息 + tool_result block
{
    "role": "user",
    "content": [
        {"type": "tool_result", "tool_use_id": "call_1", "content": "晴天 26°C"},
        {"type": "tool_result", "tool_use_id": "call_2", "content": "多云 18°C"},
    ]
}

# ===== OpenAI 格式 =====
# Assistant 消息：文本在 content，工具在 tool_calls
{
    "role": "assistant",
    "content": "我来查一下",
    "tool_calls": [
        {"id": "call_1", "type": "function", "function": {"name": "get_weather", "arguments": '{"city":"上海"}'}},
        {"id": "call_2", "type": "function", "function": {"name": "get_weather", "arguments": '{"city":"北京"}'}},
    ]
}

# Tool 结果：独立 role
{"role": "tool", "tool_call_id": "call_1", "content": "晴天 26°C"}
{"role": "tool", "tool_call_id": "call_2", "content": "多云 18°C"}
```

### 2.3 Anthropic 更通用的理由

1. **content 是列表** → 一条消息可以同时包含文本 + 多个工具调用 + 图片
2. **新增类型只需加 type** → 扩展性强（text / tool_use / tool_result / image / thinking / code_execution）
3. **User 和 Assistant 结构对称** → 都是 `[block, block, ...]`
4. **OpenAI 是历史原因** → 纯文本起步，函数调用是后加的补丁

## 3. Python 参数传递与打包解包

### 3.1 可变 vs 不可变对象

```python
# 可变对象（list, dict, set, 类实例）
# 行为像「引用传值」——修改会反映到外部
def add_item(lst, item):
    lst.append(item)  # 会影响外部

my_list = [1, 2]
add_item(my_list, 3)
print(my_list)  # [1, 2, 3]

# 不可变对象（int, str, tuple, float）
# 行为像「传值」——重新赋值不会影响外部
def set_value(x):
    x = 100  # 只是重绑局部变量

n = 5
set_value(n)
print(n)  # 5
```

### 3.2 赋值 vs 修改

```python
# 赋值（=）永远只是重绑局部变量名
def reassign(lst):
    lst = [4, 5, 6]  # 局部变量 lst 指向新对象，外部不受影响

# 修改操作会影响外部
def mutate(lst):
    lst.append(4)      # ✅
    lst[0] = 99        # ✅
    lst.extend([7,8])  # ✅
```

### 3.3 在 c2.py 中的副作用问题

```python
# c2.py 原始写法：messages.append 写在 llm() 内部
def llm(messages):
    messages.append(...)  # 副作用！函数耦合了 API 调用和历史维护
    response = client.messages.create(messages=messages)
    ...
    
# 违反单一职责：一个函数既要调用 API，又要管理消息历史
# 难以测试：调用函数会修改外部传入的列表
```

### 3.4 `*args` 和 `**kwargs` —— 打包与解包

```python
# ===== 定义时：打包 =====
def f(*args, **kwargs):
    # *args   → 多余的位置参数打包成 tuple
    # **kwargs → 多余的关键字参数打包成 dict
    print(args)    # (1, 2, 3)
    print(kwargs)  # {"x": 10, "y": 20}

f(1, 2, 3, x=10, y=20)

# ===== 调用时：解包 =====
def g(a, b, c):
    print(a, b, c)

nums = [1, 2, 3]
g(*nums)  # 解包列表 → 1 2 3

data = {"a": 10, "b": 20, "c": 30}
g(**data)  # 解包字典 → g(a=10, b=20, c=30)
```

### 3.5 在 c3.py 中的 Dispatcher 模式

```python
# handler(**block.input)   ← ** 解包
# block.input = {"path": "a.py", "limit": 10}
# handler(path="a.py", limit=10)  ← 等价于

# 每个 handler 签名不同，但通过 **kw 适配
TOOL_HANDLERS = {
    "bash":       lambda **kw: run_bash(kw["command"]),
    "read_file":  lambda **kw: run_read(kw["path"], kw.get("limit")),
    "write_file": lambda **kw: run_write(kw["path"], kw["content"]),
    "edit_file":  lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
}
```

### 3.6 Python 中 `*` 的所有用途

| 用法 | 示例 | 说明 |
|------|------|------|
| 函数定义打包 | `def f(*args)` | 多余位置参数 → tuple |
| 函数定义打包 | `def f(**kwargs)` | 多余关键字参数 → dict |
| 函数调用解包 | `f(*list)` | 列表 → 位置参数 |
| 函数调用解包 | `f(**dict)` | 字典 → 关键字参数 |
| 赋值解包 | `a, *b = [1,2,3]` | a=1, b=[2,3] |
| 字面量展开 | `[*a, *b]` | 合并列表 |
| 字面量展开 | `{**d1, **d2}` | 合并字典 |
| 强制关键字 | `def f(a, *, b)` | b 只能用关键字传 |

## 4. Tool 系统设计

### 4.1 架构模式

```
Claude 返回 content block
        │
        ▼ 遍历 response.content
   block.type == "tool_use" ?
        │
        ▼ 是
   从 TOOL_HANDLERS 字典按 block.name 查找 handler
        │
        ▼
   handler(**block.input) ← ** 解包 input dict 为关键字参数
        │
        ▼
   返回结果 → 构造 tool_result block → 追加到 messages
        │
        ▼
   继续循环（直到 stop_reason != "tool_use"）
```

### 4.2 agent_loop 有限状态机

```
         ┌──────────────┐
         │  发请求给 LLM  │
         └──────┬───────┘
                │
                ▼
      stop_reason 是什么？
                │
    ┌───────────┼───────────┐
    │           │           │
    ▼           ▼           ▼
 "tool_use"  "end_turn"  其他（错误）
    │           │           │
    ▼           ▼           ▼
 执行工具    返回结果    处理错误
    │
    ▼
 追加结果到
 messages
    │
    └──────────→ 继续循环
```

### 4.3 工具定义三要素

每个工具需要三个东西：

```python
# 1. 实现函数
def run_read(path: str, limit: int = None) -> str:
    ...

# 2. Schema 定义（告诉 Claude 这个工具长什么样）
{"name": "read_file", "description": "Read file contents.",
 "input_schema": {
     "type": "object",
     "properties": {
         "path": {"type": "string"},
         "limit": {"type": "integer"}
     },
     "required": ["path"]
 }}

# 3. Handler 注册（运行时分发）
TOOL_HANDLERS["read_file"] = lambda **kw: run_read(kw["path"], kw.get("limit"))
```

**加一个新工具 = 写函数 + 写 schema + 挂 handler，三者互不干扰。**

## 5. Content Block 类型与协议

### 5.1 Anthropic Content Block 类型

| 类型 | 发送者 | 说明 |
|------|--------|------|
| `text` | user / assistant | 纯文本内容 |
| `image` | user | 图片（base64 或 URL） |
| `tool_use` | assistant | 模型想调用工具 |
| `tool_result` | user（Agent） | 工具执行结果 |
| `tool_exec` | user | 工具执行信息 |
| `thinking` | assistant | Claude 的思考过程 |

### 5.2 Tool Use 与 Tool Result 的链接

```python
# Assistant 发出 tool_use（带 id）
{"type": "tool_use", "id": "toolu_abc123", "name": "bash", "input": {"command": "ls"}}

# Agent 返回 tool_result（用 tool_use_id 匹配）
{"type": "tool_result", "tool_use_id": "toolu_abc123", "content": "file1.txt\nfile2.txt"}
```

**`tool_use_id` 是联结请求和结果的桥梁。** Agent 必须把 Claude 给的 id 原样返回。

## 6. MCP 集成概念

### 6.1 MCP 解决什么问题

```
c3 手动流程：
写函数 → 写 schema dict → 写 lambda → 加工具改 3 处

MCP 自动流程：
MCP Server 声明工具 → Agent 通过 MCP Client 发现工具 → 零改动加工具
```

### 6.2 完整流程

```
Agent（MCP Client）              MCP Server（独立进程）
      │                               │
      │  stdio / HTTP 连接             │
      │──────────────────────────────→│
      │                               │
      │  list_tools()                 │
      │──────────────────────────────→│
      │                               │── 返回 tools 列表（schema）
      │←──────────────────────────────│
      │                               │
      │  （自动注册到 TOOLS / TOOL_HANDLERS）
      │                               │
      │  call_tool("get_weather", {city:"上海"})
      │──────────────────────────────→│
      │                               │── httpx.get("http://123.com/weather")
      │                               │
      │←──────────────────────────────│
      │  返回结果                      │
```

### 6.3 标准化协议

| | c3 手动 | MCP |
|---|--------|-----|
| 工具发现 | 手动写 schema | `list_tools()` 自动获取 |
| 工具调用 | 自己写 handler | `call_tool(name, args)` |
| 新增工具 | 改 3 处代码 | Server 加一个 `@app.tool()` |
| 多个来源 | 需要自己合并 | 可同时连多个 Server 自动合并 |

## 7. SkillLoader 知识分层（c4）

### 7.1 解决的问题

**System Prompt 有长度限制**，不能把所有知识都塞进去。

### 7.2 两层知识注入

```
第一层（system prompt）：技能目录
  ┌──────────────────────────────────────────┐
  │ Skills available:                         │
  │   - code-review: Perform thorough code    │
  │     reviews with security, performance,   │
  │     and maintainability analysis.         │
  │   - python-debug: Debug Python projects   │
  └──────────────────────────────────────────┘
  Claude 看到目录但不知道详情
        │
        ▼ 需要时调用 load_skill(name="code-review")
        │
第二层（按需加载）：完整技能正文
  ┌──────────────────────────────────────────┐
  │ # Code Review Skill                       │
  │ ## Security                               │
  │ - Injection vulnerabilities               │
  │ - Authentication issues                   │
  │ ...                                       │
  └──────────────────────────────────────────┘
```

### 7.3 SkillLoader 实现

```python
class SkillLoader:
    def __init__(self, skills_dir: str):
        self.skills_dir = Path(skills_dir)
        self.skills = {}
        self._load_all()
    
    def _load_all(self):
        # 递归搜索所有 SKILL.md
        for f in sorted(self.skills_dir.rglob("SKILL.md")):
            text = f.read_text()
            meta, body = self._parse_frontmatter(text)
            # YAML frontmatter 定义元信息
            name = meta.get("name", f.parent.name)
            self.skills[name] = {"meta": meta, "body": body}
    
    def get_descriptions(self) -> str:
        # 生成轻量目录（给 system prompt）
        lines = [f"  - {n}: {s['meta']['description']}" 
                 for n, s in self.skills.items()]
        return "Skills available:\n" + "\n".join(lines)
    
    def load_skill(self, name: str) -> str:
        # 按需加载完整内容
        return self.skills[name]["body"]
```

### 7.4 SKILL.md 格式

```markdown
---
name: code-review
description: Perform thorough code reviews with security, performance, and maintainability analysis.
---

# 技能正文
这里放完整的技能知识...
```

## 8. Subagent 任务分层（c5）

### 8.1 解决的问题

**单个 Agent 处理复杂任务时，历史上下文越来越长**，token 消耗大。

### 8.2 核心设计

```python
# 工具分两级
CHILD_TOOLS = [
    {"name": "bash", ...},
    {"name": "read_file", ...},
    {"name": "write_file", ...},
    {"name": "edit_file", ...},
]  # 子 Agent 没有 subagent，防止无限嵌套

PARENT_TOOLS = CHILD_TOOLS + [
    {"name": "subagent", ...}
]  # 父 Agent 可以派生子任务

# agent_loop 参数化，父子共用
def agent_loop(system: str, messages: list, tools: list):
    ...
```

### 8.3 subagent 的执行逻辑

```python
def run_subagent(prompt: str, description: str) -> str:

    sub_messages = [{"role": "user", "content": prompt}]
```

### 8.4 subagent vs subprocess

| | subprocess | subagent |
|---|-----------|---------|
| **说哪种语言** | shell 命令 | 自然语言 |
| **输入** | `"ls -la"` | `"分析这个项目的架构"` |
| **输出** | stdout/stderr 原始文本 | 自然语言总结 |
| **有没有脑子** | ❌ 机器执行 | ✅ 有 LLM 能自主决策 |
| **复杂度** | 单一原子操作 | 一系列操作组合 |
| **内部可能做什么** | 执行一个命令 | 调用多次 bash/read_file/write_file |

**subprocess 是手和脚，subagent 是一个能独立干活的小人。subagent 内部会调用 subprocess 来执行命令。**
